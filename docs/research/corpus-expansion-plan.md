# Corpus Expansion Plan — unblocking B2 / B3 / B4

Status: research runbook, no corpus generated by this document.
Written: 2026-06-13.
Scope: a cross-session, skill-driven plan to grow the labeled corpus so the
slice (B2), robustness (B3), and language-threshold-calibration (B4) work has
enough data to be meaningful. Reuses the existing metadata-first rebaseline
pipeline; it does not introduce a new corpus store or a new privacy model.

> Patina's bar is meaning-preserving, auditable AI-style-signal reduction — NOT
> detector bypass. Every collection step here is for measurement and
> calibration evidence, never for evasion.

## Why this is needed now

The measurement infrastructure shipped (`#A1`/`#B1` and `#B2`/`#B3`), but the
labeled corpus is too small for two of the three to deliver value:

| Capability | Shipped | Data blocker |
|---|---|---|
| B2 slice metrics (`tests/quality/slice-metrics.mjs`, report `## Slice metrics`) | yes | Most slices show `insufficient_data`; only `language`/`class`/`lengthBucket` have real n. `domain`/`register`/`generator`/`edited` are all `unspecified`. |
| B3 robustness report (`scripts/robustness-report.mjs`, `docs/benchmarks/robustness-latest.md`) | yes | Works today (transforms the 49 fixtures) but a wider corpus makes the retention rates representative instead of anecdotal. |
| B4 language threshold calibration | NOT shipped (blocked) | The baseline benchmark is at **100% / ROC·PR-AUC 1.000** on 49 fixtures and each language has only ~5–6 negatives, so there is **no calibration signal** and a 1% FPR target floors to 0. |

The current suspect-zone fixtures carry only `language`, `class`,
`expected_hot`, and a free-form `topic` (no `domain`/`register`/`generator`/
`edited`). The metadata-first rebaseline corpus (`artifacts/rebaseline-2025/`)
already carries the richer fields — this plan grows that corpus and wires its
metadata into the B2/B4 schema.

## Target matrix

Cells are `language × class × register × model_family`. Minimum counts are tied
to the consumers: B2 marks a slice `supported` at **n ≥ 5**
(`DEFAULT_MIN_SLICE_COUNT`); B4's 1% FPR target only becomes meaningful at
**≥ 100 negatives per language** (below that, report strict zero-FP diagnostics,
never a tuned threshold). The existing public claim gate is **n ≥ 100 per claim
cell**.

| Language | natural-human (negatives) | AI-like (positives) | edited-AI | Status |
|---|---:|---:|---|---|
| KO | ≥100 (250 web rows exist, 5 registers × 50) | ≥100 per GPT/Claude/Gemini/open-weight cell | ≥1 light + ≥1 heavy per register, grow to ≥30 | negatives strong; positives + edited-AI thin |
| EN | ≥100 across ≥4 registers | ≥100 per model-family cell | same | rebaseline rows exist; verify register spread |
| ZH | ≥100 across ≥3 registers | ≥100 per model-family cell | same | near-zero public coverage — pilot first |
| JA | ≥100 across ≥3 registers | ≥100 per model-family cell | same | near-zero public coverage — pilot first |

Registers (reuse the KO set): `academic-summary`, `product-doc`,
`chat-update`, `blog`, `technical-how-to`. Extend per language only with a
reviewer note.

## Metadata schema → B2 slice dimensions

Collect with the existing manifest fields
(`humanization-data-backlog.md#manifest-fields`); the table below maps them onto
B2's `SLICE_DIMENSIONS` so collected data flows straight into the slice report
once fixtures/manifests carry the fields. Decide where each lives: redistributable
text → fixture frontmatter; restricted text → public manifest (hash-only).

| B2 slice dimension | Manifest / frontmatter field | Allowed values | Missing-value behavior |
|---|---|---|---|
| `language` | `lang` | ko, en, zh, ja | required |
| `class` | `class` | ai-like, natural-human, lightly-edited-ai, heavily-edited-ai | required |
| `register` | `register` | the 5 registers above (+ reviewed extras) | `unspecified` |
| `domain` | `domain` (new, optional) | freeform reviewed tag | `unspecified` |
| `generator` | `model_family` | gpt, claude, gemini, open-weight, human | `unspecified` |
| `lengthBucket` | derived (`lengthBucket()` on body code points) | empty/short/medium/long | derived, never stored |
| `edited` | `edit_depth` | none, light, heavy | `unspecified` |

Maintainer decisions still required before B2 slices and B4 calibration can make
claims (these are the four-per-story unblock gates from the approved plan):

- **B2**: corpus source/manifest with enough records per slice; required
  metadata fields + allowed values + where they live; privacy/license per source;
  minimum slice-count rule for "reportable vs insufficient".
- **B4**: operating policy (target: TPR@1%/5%FPR · precision floor · recall
  floor · or preserve current decisions); data sufficiency (min negatives per
  calibrated language, else strict zero-FP diagnostics); approved before/after
  deltas by language incl. must-not-regress slices; public messaging wording.

## Skill-driven, multi-session workflow

Run these in separate sessions; each writes durable artifacts other sessions
read.

1. **`/skill:deep-interview`** — turn the open maintainer decisions above
   (B2 schema/min-count, B4 operating policy + allowed deltas + messaging) into a
   spec under `.gjc/specs/`. This is requirements-only; it must not touch product
   code or thresholds.
2. **`/skill:ralplan`** — plan the collection itself: per-language/per-source
   waves, intake order, and the verification gate. Stop at pending approval.
3. **`/skill:ultragoal`** — durable ledger across sessions. One goal per
   collection cell (e.g. "ZH AI-like × product-doc → n≥100"); checkpoint each as
   counts reach target so progress survives session boundaries.
4. **`/skill:team`** — optional: parallelize collection across languages/sources
   with workers + worktrees when the cells are independent.

Each session uses the existing pipeline, e.g.:

```bash
# Hash-only public-web human controls from a tracked source inventory
npm run benchmark:rebaseline:web -- \
  --input artifacts/rebaseline-2025/sources.<lang>-public.jsonl \
  --output artifacts/rebaseline-2025/private/<lang>-human-controls.generated.private.jsonl \
  --target-per-register 50

# Controlled model-era positives from repo-owned prompts
npm run benchmark:rebaseline:generate-modern -- --lang <lang> --register <register>

# Normalize + split public/private, then score without committing raw text
npm run benchmark:rebaseline:intake -- --input <intake>.jsonl --require-source-review
npm run benchmark:rebaseline:score  -- --input <private>.jsonl --output <manifest>.public.jsonl
npm run benchmark:rebaseline -- --input <manifest>.public.jsonl --json

# Promote redistributable natural-human rows into suspect-zone fixtures
npm run benchmark:rebaseline:fp-fixtures -- --dry-run
npm run benchmark:ranges   # refresh expected-ranges.json, review the diff
```

## Exit / unblock gates

- **B2 becomes valuable** when ≥1 metadata dimension beyond language/class/length
  has ≥2 buckets each with n ≥ 5 reviewed rows (slices stop reading
  `insufficient_data`/`unspecified`). No threshold change.
- **B3 stays report-only**; rerun `npm run benchmark:robustness` after fixtures
  grow so retention rates are representative. No threshold change.
- **B4 becomes ready** only when ALL hold: (a) the four maintainer decisions are
  recorded; (b) per-calibrated-language negatives ≥ the agreed minimum (≥100 for
  a 1% FPR target, else strict zero-FP only); (c) AI-like + edited-AI positive
  controls exist per language; (d) the benchmark stops being perfectly separable
  (i.e., there is a real FP/FN trade-off to calibrate). Until then, B4 stays
  blocked and thresholds are unchanged.

## Guardrails (carry over)

- Do not frame collection as detector bypass or evasion.
- Do not commit private or no-redistribution raw text — hashes/metadata/aggregates only.
- Do not tune a global threshold to fix one language/model-family miss cluster.
- Do not blend learner-Korean (or other stress envelopes) into native-human baselines.
- Do not publish broad ZH/JA claims from pilot data until the n ≥ 100 claim gate is met.
- Do not change detector thresholds (B4) outside an approved-delta, reviewed PR.
